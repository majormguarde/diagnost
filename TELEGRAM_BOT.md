# Telegram-бот и связь с клиентами (Diagnost)

Бот обслуживает **те же сценарии**, что и веб-приложение: привязка аккаунта по коду из кабинета, команда **`/zakaz КОД`** для получения текста заказ-наряда, команда **«Мои заказы»**.

Обработка сообщений реализована в коде приложения:

- `app/telegram_handlers.py` — разбор входящих **Update** и ответы через Bot API  
- `app/telegram_bot.py` — токен/имя бота, `sendMessage`, выдача и погашение кодов наряда  
- `app/telegram_poller.py` — **long polling** в фоновом потоке (опционально)

## 1. Создать бота в Telegram

1. Откройте [@BotFather](https://t.me/BotFather) → **/newbot** → задайте имя и username (оканчивается на `bot`).  
2. Скопируйте **HTTP API token**.  
3. В админке: **Связь** → поля **«Имя Telegram-бота»** (username без `@`) и **«Токен бота»** → сохранить.  
   Либо задайте переменные окружения `TELEGRAM_BOT_NAME` и `TELEGRAM_BOT_TOKEN` (значения из БД имеют приоритет).

## 2. Режим получения обновлений: `polling` или `webhook`

Переменная окружения **`TELEGRAM_UPDATES_MODE`**:

| Значение   | Поведение |
|-----------|-----------|
| `polling` | **По умолчанию.** При старте `run.py` / приложения поднимается **daemon-поток**, который вызывает `getUpdates`. Удобно на локальной машине **без HTTPS**. При первом запросе вызывается **`deleteWebhook`**, чтобы не было конфликта с ранее настроенным вебхуком. |
| `webhook` | Фоновый polling **не** запускается. Обновления только на **POST** `https://<ваш-домен>/telegram/webhook` (нужен **HTTPS** и открытый URL). В BotFather: **/setwebhook** на этот адрес. |

На **одном токене** не используйте одновременно активный webhook и polling: Telegram отдаст обновления только одному способу.

Примеры `.env`:

```env
# Локальная разработка (бот в одном процессе с Flask)
TELEGRAM_UPDATES_MODE=polling
TELEGRAM_BOT_TOKEN=123456:ABC...
TELEGRAM_BOT_NAME=MyDiagBot

# Продакшен за reverse-proxy с HTTPS
TELEGRAM_UPDATES_MODE=webhook
```

В режиме **`webhook`** в админке **Связь** укажите **«Публичный URL сайта»** — там же показывается готовый URL вебхука для BotFather.

## 3. Клиенты: привязка и заказ-наряды

1. Клиент в **личном кабинете** → раздел Telegram → **сгенерировать код** привязки.  
2. В чате с ботом: **`/start <код>`** — создаётся связь `TelegramLink` (chat_id ↔ user_id).  
3. Сервис на печатной форме наряда нажимает **Telegram** — создаётся код **`/zakaz …`**; при наличии токена бот **отправляет** клиенту сообщение с ссылкой на бота и командой (если привязка уже есть).  
4. Клиент пишет боту **`/zakaz КОД`** (или только 12 символов кода) — бот присылает текст наряда (одноразовый код).

## 4. Запуск вместе с приложением

```bash
python run.py
```

При **`TELEGRAM_UPDATES_MODE=polling`** и заданном токене в логах появится строка о запуске long polling. Дополнительных процессов не требуется — поток **daemon** завершится вместе с процессом Python.

При **`FLASK_DEBUG=1`** (встроенный reloader) поток polling поднимается **только в рабочем дочернем процессе**, чтобы не было двух poller на одном токене.

Проверка без Telegram-клиента:

```http
GET /telegram/status
```

Ответ JSON: `bot_configured`, `bot_name`, `updates_mode`.

## 5. Безопасность

- Токен бота в БД / `.env` — **не** коммитьте в git.  
- URL вебхука должен быть **HTTPS** (требование Telegram).  
- CSRF для `/telegram/webhook` отключён (`csrf.exempt`) — типично для вебхуков; при желании ограничьте доступ по IP на прокси.
